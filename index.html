기기 재고 앱 — 비개발자를 위한 이해 가이드

이 앱을 직접 고치진 못해도, 무슨 일이 일어나는지 이해하는 것이 목표입니다. 통역에 비유하면, 동시통역 부스를 직접 설계하진 못해도 부스 안 장비가 각각 무슨 역할인지는 아는 것과 같습니다.

1. 큰 그림 — 이 앱은 어떻게 돌아가나

[브라우저]  ←→  [서버(앱 프로그램)]  ←→  [데이터베이스(자료 창고)]
 화면 보고 클릭     화면을 만들어 보냄        기기·대여 기록 저장
  • 브라우저: 팀원이 equip-inventory-gilt.vercel.app에 접속해 보는 화면 (크롬 등)
  • 서버: 그 화면을 만들어 브라우저에 보내주는 프로그램. 우리 코드가 여기서 돌아감
  • 데이터베이스: "기기 31번 사용 불가", "누가 무엇을 빌렸다" 같은 실제 데이터 저장소

핵심: 화면(디자인)과 데이터(기록)는 분리돼 있습니다. → 버튼 색을 바꿔도 데이터는 그대로, 대여로 데이터가 바뀌어도 디자인은 그대로.

2. 등장하는 "도구"들

이름정체비유
프로젝트 폴더
Desktop\equip-inventory		앱의 설계도(코드)가 든 폴더통역 자료가 든 서류함
터미널글자로 명령을 내리는 검은 창타이핑으로 시키는 비서
GitHub코드 백업 + 변경 이력 창고(인터넷)구글 드라이브 + 수정 기록
Vercel코드를 인터넷에 띄워주는 서비스(=배포)완성본을 게시판에 게시
Supabase데이터베이스 = 실제 자료 저장소자동화된 자료 창고(엑셀 대체)
  • 터미널: npm run build, npx vercel --prod 같은 명령을 붙여넣는 그 창. 마우스 대신 글자로 명령.
  • GitHub: 고칠 때마다 "뭘 바꿨다"는 기록(커밋)을 남기고 인터넷 창고에 올림(푸시). 컴퓨터가 고장나도 안전.
  • Vercel = 배포: 고친 코드는 처음엔 내 컴퓨터에만 있음. 팀원이 보려면 인터넷에 올려야 하는데 그게 배포. Vercel이 담당.
  • Supabase: 예전 구글 스프레드시트 역할. 화면의 "사용 가능 30, 사용 불가 1" 숫자는 여기서 실시간 계산.

3. 앱을 만든 "재료" (기술 스택)

재료역할비유
Next.js / React화면을 조립하는 틀(프레임워크)화면을 레고 블록처럼 조립
TypeScript앱의 동작을 적는 프로그래밍 언어실제 대본을 쓰는 언어
Tailwind CSS디자인(색·크기·간격)을 입히는 도구옷의 색·핏 스타일링
  • 컴포넌트: 화면을 작은 조각(블록)으로 나눠 만듦. note-row.tsx(비고 한 줄), status-section.tsx(기기 현황) 등이 각각 한 블록.
  • Tailwind 예시: text-3xl(글자 크게), font-medium(굵기), px-5 py-2(여백), bg-brand(민트 배경), rounded-xl(둥근 모서리).

4. 파일 지도 — 어디 보면 뭐가 있나

equip-inventory\app 폴더 안이 화면들. 파일 이름 ≈ 화면/조각.

파일무슨 화면 / 역할
page.tsx홈 화면(메인). "쓰고있어요!", "쓸래요!", 위치별 기기 표
status-section.tsx홈의 기기 현황 카드 + 이어폰·마이크 패널
locker-section.tsx홈의 사물함 영역 ("팀 사물함엔 뭐가 있더라?")
notes-preview.tsx홈의 "기기 비고" 팝업
checkout/page.tsx대여하기 화면
notes/page.tsx기기 비고 편집 화면
note-row.tsx비고 편집 화면의 비고 한 줄(사용 불가 토글)
layout.tsx모든 화면 공통 설정(탭 이름, 폰트)
globals.css전역 디자인 규칙(브랜드 민트색, 깜빡임 애니메이션)
lib/화면이 아니라 데이터를 가져오는 뒷일 담당
db/데이터베이스 구조 설계도(SQL)

예: "쓸래요! 버튼 색" → page.tsx, "탭 이름" → layout.tsx, "깜빡임" → globals.css.

5. 수정 한 번이 화면에 뜨기까지

① 요청        : "쓸래요 버튼 색 바꿔줘"
      ↓
② 코드 수정   : Claude가 .tsx 파일을 고침
      ↓
③ 빌드        : npm run build   (오류 검사 + 완성본 만들기)
      ↓
④ 커밋/푸시   : git commit / push   (변경 기록 + GitHub 백업)
      ↓
⑤ 배포        : npx vercel --prod   (인터넷에 올림)
      ↓
⑥ 사이트에 반영 ✅  (equip-inventory-gilt.vercel.app)
  • 빌드(build): 코드를 컴퓨터용 완성본으로 변환 + 오류 검사. "인쇄 전 오탈자 검수 + 제본".
  • 배포(deploy): 완성본을 인터넷에 띄우기. 이게 끝나야 팀원 화면에 보임.
  • 커밋/푸시: 백업·이력 남기기. 화면엔 영향 없지만 안전장치.
내 역할은 ⑤(배포 명령 복붙)뿐. ②③④는 Claude가 합니다.

6. 용어 미니 사전

용어뜻
배포 / deploy코드를 인터넷에 올려 실제로 쓰게 하기
빌드 / build코드를 검사하고 실행용 완성본으로 변환
커밋 / commit"이번에 뭘 바꿨다"는 한 덩어리의 기록
푸시 / push그 기록을 GitHub 창고에 업로드
컴포넌트화면을 이루는 재사용 가능한 조각(블록)
클라이언트/서버클라이언트=사용자 브라우저 쪽 / 서버=뒤에서 처리하는 쪽
마이그레이션데이터베이스 구조를 바꾸는 작업(SQL)
env(환경변수)비밀번호·접속 정보를 코드 밖에 숨겨두는 설정
TLS 인증서회사망 보안검사 때문에 생기는 문제를 우회하는 명령(배포 때 늘 붙임)

7. 왜 디자인 조정이 미세하게 여러 번 필요한가

버튼 하나에도 조절 항목이 따로따로 많습니다: 글자 크기, 굵기, 폰트 종류, 색, 여백, 모서리, 테두리, 배경. "조금 못생겼다"의 원인이 그중 무엇인지(예: 폰트가 원래 굵기 하나뿐이라 못 줄임) 찾아 하나씩 바꾸는 과정. 디자인은 원래 이런 미세 조정의 연속입니다.

부록: 배포 명령 (참고용)

프로젝트 폴더를 연 터미널에서 순서대로:

npm run build

(빌드 성공 확인 후)

$env:NODE_TLS_REJECT_UNAUTHORIZED="0"
npx vercel --prod
